> Classes v. Modules
July 13, 2014

Modules and Classes are both Object-Oriented Programming tools. Both modules and classes store methods, and they look fairly similar. Modules, however, do not create object, but rather they simply define behavior. Modules, therefore, can be thought of like libraries of beviours, or chunks of reusable code, that can be called on to create a program.

In my research of Modules and Classes, I found two keywords that came up over and over for Modules: Namespacing and Mixins.

  • Namespacing does kind of what it sounds like - it creates separate spaces for naming. Ok, maybe that doesn't sound like anything. What I mean is that you can have a method of the same name stored in different modules (the spaces) without causing confusion. Ruby Monk compares the use of Modules to a filesystem in which different files of the same name can be stored in different folders. The example I liked the best is from the Pragmatic Programmer's Guide. This guide uses the example in which the method #sin could be defined in a module Trig of trigonomic functions and another method #sin could be defined in a module Action of good and bad (sins) behaviors. If the two #sin methods are stored in separate spaces, they can be called according to the modules using Trig.sin and Action.sin

  • Mixins allow for modules to be included (by using the word "include") within Classes. This is where the library or toolbox or filing cabinet (however you want to think of it) of re-usable code comes in to play. You can create a Module that stores a set of actions and then invoke them ("include" them) when defining a Class. The object(s) created by that Class will then be able to perform the actions stored in that Module. Even though its frowned upon, this allows for multiple inheritances.
Example Code:

Let's look at a block of code that utilizes the mixin property of Modules within Classes. In our example, we'll define a Module called Wash. Within this Module, we store a number of methods that are used when cleaning, such as rinse and lather

module Wash

    def rinse
      puts "run in water for at least 10 seconds"
    end

    def lather
      puts "scrub with soap for at least 15 seconds"
    end
end

Now, we can apply this module to two separate classes, to which we could imagine that both the rinse and lather methods could be applied. Lets wash a dish and a car - which are two objects with very different properties and uses, but two which these methods are relevant.

class Car

    include Wash
end

class Dish

    include Wash
end

In this example, the module and the class to which its applied would have to be in the same file. The module could be stored in a separate file, and in that case you would have to write

require "wash"

in order to reference the module. If the module were stored in a separate folder than the class you would write

$Load_Path << "."
require "wash"

Modules allow for an added level of complexity to a program and reduce the amount of repetative work that needs to be done
