> What is Rails?
July 24, 2014

Rails is the short-hand name for "Ruby on Rails". While Ruby is the programming lanaguage that runs on Rails, it is not to be confused with the Rails web application development framework that follows the model-view-controller pattern. Rails makes programming web applications easier and more efficient because it comes equipped with a framework common to the development of most web applications. It therefore reduces the amount of initial development required by the developer.

> Terms: Web Application Development Framework and Model-View Controller

What is a Web Development Framework?

The definition straight from Wikepedia defines it as: "a software framework that is designed to support the development of dynamic websites, web applications, web services and web resources. The framework aims to alleviate the overhead associated with common activities performed in web development. For example, many frameworks provide libraries for database access, templating frameworks and session management, and they often promote code reuse."

This is a pretty succinct definition that discusses what Rails is and does and a few of its key components. When we take a closer look at Rails below, we'll talk about what the libraries, templating frameworks, and session management provided in the framework actually look like.

What is the Model-View-Controller Pattern?

The Model-View-Controller pattern basically dicates how the web application framework is designed. Again, from Wikepedia: "Model-view-controller (MVC) is a software architectural pattern for implementing user interfaces. It divides a given software application into three interconnected parts, so as to separate internal representations of information from the ways that information is presented to or accepted from the user."

In some ways you can think of the Model-View-Controller pattern as a division of labor with specified roles and responsibilities. In the visual representation below, the roles, relationships, and interactions are depicted in relation to the user.

  • The controller is basically the Manager. It takes user inputs, data submissions and basically everything that comes in to the application through the browser, and decides what to do with it. It communicates messages to the model and then returns what it receives from the model back to the user.
  • The model manages the application's data, logic, and rules. It takes in the code and gives you something back - this requires a whole process of interacting with the database to store and validate the data (all of which is dictated by the web application framework). The model is what does the heavy-lifting.
  • The view is the output from the model to the user, presented using HTML, CSS, and JavaScript.
  • The user interacts with the application and has the ability to input and retrieve information.

> What does the Rails framework look like?

The Rails Guide walks you through how to make a simple web application using Rails and it is worth at least reading through to get a better idea of how Rails works. I'll be using this tutorial to explain what the Rails web development framework looks like and how you interact with it to get a new app up and running.

First, when you create a new application, Rails will auto-generate a number of files and folders. These pre-designed documents and the structure in which they reside (the folders) are the web application ramework of Rails. The basic structure includes the following files and folders:

  • app/
    includes the models (app/models/), views (app/views/), and controllers (app/controllers/). This is where the developer does the majority of their work.
  • bin/
  • config/
  • config.ru
  • db/
  • Gemfile; Gemfile.lock
  • lib/
  • log/
  • public/
  • Rakefile/
  • README.doc
  • test/
  • tmp/
  • vendor/

The following are the basic steps for setting up a new app. I will show the basic commands without much explanation so that you can get an idea of how some of the files and folders in the framework are being used.

  1. Create a new app called blog
    $rails new blog

  2. To get the app running, start the server
    $bin/rails server

  3. Create a controller and a view called welcome
    $bin/rails generate welcome index

    The following will then run in the command line, creating a number of files in your app/ folder:
    create app/controllers/welcome_controller.rb
    route get 'welcome/index'
    invoke erb
    create app/views/welcome
    create app/views/welcome/index.html.erb
    invoke test_unit
    create test/controllers/welcome_controller_test.rb
    invoke helper
    create app/helpers/welcome_helper.rb
    invoke test_unit
    create test/helpers/welcome_helper_test.rb
    invoke assets
    invoke coffee
    create app/assets/javascripts/welcome.js.coffee
    invoke scss
    create app/assets/stylesheets/welcome.css.scss

  4. To set up your homepage, open the config/routes.rb file in a text editor. Uncomment the line that says root 'welcome#index' so that it looks like:
    Rails.application.routes.draw do
    get 'welcome/index'
    # The priority is based upon order of creation:
    # first created -> highest priority.
    #
    # You can have the root of your site routed with "root"
    root 'welcome#index'
    #

  5. To begin working on your homepage, open the index file (app/views/welcome/index.html.erb) in a text editor. You can begin by typing basic HTML that will present on the homepage.
    <h2><font color="red">Check out my awesome new webpage!</font></h2> If you launch the server again and navigate to http://localhost:3000 in your browser, the HTML that you entered should appear on the page like so:

    Check out my awesome new webpage!

> The Other Frameworks

Rails is not the only web application development framework for Ruby, but it is the most popular. Other frameworks include: Sinatra, Merb, and Camping. To gain even futher insigths into what Rails is, it may be helpful to research some of these other frameworks and how they differ from Rails. Another suggestion is to research, for example, the different frameworks for running Java, as there are many more commonly used Java frameworks, each with their own design.
